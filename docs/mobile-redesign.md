## 1. 핵심 컨셉 ✅

**"아저씨(스마트폰 서툰 사원)도 배우지 않고 바로 쓰게."**

화려함보다 실수 없이 명확하게가 목표. 실제 사용자는 아버지 회사 내부 직원 ~10명, 성장 없음(고정).

설계 원칙:
- **큰 글씨(본문 16px+)·큰 버튼(탭 타깃 44px+)** — 노안·서툰 손가락 배려
- **버튼에 글자를 붙인다** — `＋` 아이콘만 X → `＋ 새 일정`처럼 이름을 붙여 뭘 하는지 바로 인지
- **숨은 제스처 없음** — 스와이프 삭제·길게 누르기 안 씀. 보이는 것만 누르면 됨
- **한 화면에 한 가지** — 정보 욱여넣지 않음. 나머지는 '더보기'로 치움
- **돌아가는 길 항상 있음** — 상세 화면 좌상단 `‹ 뒤로` 고정
- **상태는 색+글자 이중 표시** — 예정/진행중/지연/완료를 색약·노안도 구분

## 2. 운영 모델 (누가 입력하나) 🤔

> 이게 화면보다 먼저 정해야 하는 근본 문제. "모두가 매일 자기 일정을 성실히 입력"하는 모델은 **비현실적** → 그렇게 설계하지 않는다.

### 실제 일하는 모양새
회사가 작아 역할이 칼같이 안 나뉜다("니일 내일 없이"). 그래도 큰 틀은:

| 역할 | 스마트폰 활용 | 하는 일 |
|---|---|---|
| **본사/대표** | 가끔 조망 | 현장 만들고 담당 배정, 전체 현황 보기 |
| **현장 사수(대장)** | 주 작성자 | 자기 현장 일정 짜고 작업자 배정 |
| **작업자(아저씨)** | 거의 조회만 | 내 오늘 일 확인 + 완료 체크 ✓ |

- 현장 = 앱의 `프로젝트` 1개, 사수 = 그 프로젝트의 대장.
- 부담이 현장별로 쪼개져서 각자 자기 몫만 → 아저씨들이 안 질리는 구조.

### 입력 부담을 줄이는 2대 원칙 ✅
1. **입력 안 해도 유용해야 계속 쓴다** — 앱 열면 "오늘 나 뭐 하지"가 바로 보이는 것만으로 가치. + **푸시 알림**으로 앱 안 열어도 정보 수신(입력 아니라 수신 중심).
2. **사원에게 기대하는 액션은 "탭 한 번" 수준만** — 타이핑 요구 X. 완료 체크(✓), 현장 사진 한 장 첨부 정도.

작성자(대표·사수) 부담도 경감: **반복 일정 · 템플릿/복제**로 대량 생성.

## 3. 권한 모델 ✅ (느슨하게)

> 10명이 서로 믿고 니일 내일 없이 하는 회사 → **권한을 빡세게 나누지 않는다.** "현장 단위로 수정 잠금" 같은 벽은 과설계이자 불편. (인프라 right-sizing과 같은 논리.)

**딱 2단계로 충분:**

| 단계 | 누구 | 할 수 있는 것 |
|---|---|---|
| **직원** (로그인한 사람 누구나) | 사수·작업자 | 일정·현장 **조회·생성·수정 다 가능**, 완료체크, 사진첨부 |
| **대표** (관리자) | 대표 | 위 전부 + 사원 계정 관리 + 민감한 삭제 |

**핵심 구분: 담당자(사수)는 "권한"이 아니라 "표시"다.**
- "이 현장 담당 = 김씨"라고 **보여주기만** 함
- 다른 사람도 손댈 수 있음(막지 않음)
- 담당자 지정은 **"내 일정만 보기" 필터용**으로만 씀 — 접근 차단 X

효과: 아저씨가 "권한 없네?" 막히는 일 없음 · 급할 때 아무나 대신 입력 · 홈은 여전히 "내 오늘 일정"으로 깔끔.

> ⚠️ **현 코드와 차이:** 지금은 일정/프로젝트/거래처 생성이 전부 전역 `requireAdmin`(사원은 생성 불가). 위 모델대로면 **일반 직원도 생성·수정 가능**하게 완화해야 함. (백엔드 권한까지 손질 대상)

## 4. 정보구조 / 네비게이션 🤔

**모바일 전용 레이아웃 분리** — 데스크톱 사이드바(`Layout`) 유지 + `useMediaQuery`로 모바일이면 `MobileLayout`(하단 탭바) 분기. 페이지 로직·API·store는 공유.

**구현 순서 ✅: 모바일 먼저 → 이후 웹(데스크톱) 화면 전면 개편.**
- 모바일 작업이 버려지지 않게 **3층 구조**로: `공유층(디자인 토큰·공통 컴포넌트·API·store)` + `레이아웃층(MobileLayout 탭바 vs WebLayout 사이드바, 여기만 다름)`.
- 모바일 만들 때 공통 컴포넌트를 밖으로 빼두면 → 웹 개편 때 **레이아웃만 갈아끼우고 재사용**.
- 명세와 독립적으로 **지금 병행 가능**: PWA 전환(Vite+vite-plugin-pwa) · 모바일 골격(탭바) · 공통 컴포넌트 · 디자인 토큰 · 사진/영상 인프라(S3, `<input capture>`). **⚠️ 화면별 완성 폼(일정 생성 등)은 명세 후에** — 그 전에 만들면 재작업.
- 과도기: 모바일=새 디자인 / 웹=옛 사이드바 공존 (내부 10명이라 허용).

**하단 탭 (잠정안, 언제든 교체 가능):**

| 탭 | 내용 |
|---|---|
| 홈 | 오늘 요약 — 내 일정·할 일·팀원 현황 |
| 일정 | 중심 탭. 리스트/캘린더 전환, 날짜별 그룹, ＋로 생성 |
| 프로젝트 | 진행률·상태 카드. 상세에서 관련 일정·파일 연결 |
| 더보기 | 거래처·파일함·사원관리(대표)·설정 |

> 📌 핵심 탭 구성은 **잠정**. 실제 사용 패턴 보고 언제든 교체 — 탭바만 바꾸면 되도록 설계.

역할별 홈이 갈릴 수 있음(대표=전체 현황+＋, 작업자=내 것만·단순). 🤔

> 📌 **사진 공유가 자주 일어남(→ 섹션 5)** → `사진`이 하단 탭 후보가 될 수 있음. 파일함을 '현장 사진/파일'로 격상 검토.

## 5. 현장 사진·영상 — "카톡보다 쉽게" 🤔 (핵심 기능)

> **자주 일어나는 핵심 활동.** 지금은 **카톡으로** 현장 사진·동영상을 공유 중. 앱이 이걸 흡수하려면 **카톡만큼 쉬우면서 + 카톡이 못 하는 걸** 줘야 한다.
> (마침 현재 브랜치가 `feat/s3-storage` — 저장소는 이미 준비 중.)

### 왜 카톡을 쓰나 / 어떻게 이길까
- 카톡의 강점: 열고 → 카메라 → 전송. 마찰이 거의 없음.
- 카톡의 약점: **시간 지나면 스크롤에 묻히고 원본 만료.** 어느 현장 건지 나중에 못 찾음.
- → 앱의 승부수: **현장별로 자동 정리 + 영구 보관 + 나중에 바로 찾기.**

### UI 접근 (아저씨도 쉽게)
- **큰 `📷 사진 올리기` 버튼** — 홈/현장 화면에 큼직하게. 누르면 **카메라 바로** 또는 갤러리 다중선택.
- **현장 자동 태깅** — 어느 현장에 올릴지 이미 정해진 맥락(현장 안에서 올리면 자동). 최소 탭.
- **캡션 강요 X** — 그냥 올려도 됨. 누가·언제·어느 현장은 자동 기록.
- **썸네일 격자(카톡처럼 익숙)** — 현장별 사진첩. 탭하면 크게, 좌우 스와이프로 넘김(익숙한 제스처라 OK).
- **동영상** — 인라인 재생, 업로드 진행률 표시, 용량 큰 것 대비(백그라운드 업로드 고려).
- **공유는 여전히 카톡으로 나갈 수 있게** — OS 공유 시트로 카톡 전송 가능. 앱=원본 보관·정리, 카톡=빠른 알림. **둘 다 살림(배타적 아님).**
- **홈에 "오늘 올라온 현장 사진" 노출** — 앱 열면 최신 사진이 보여서 "카톡 대신 여기 보면 되네" 학습 유도.

### 채택 전략 — 카톡을 어떻게 이기나 (강제 최소화) ✅
> 순수 "빠르게 보내기" 속도로는 카톡 못 이김(근육 기억+수신자 이미 거기 있음). 정면승부 X, **다른 축에서** 이긴다. 강제는 최후수단(반발·우회 유발).

비-강제 지렛대 3개:
1. **"못 찾는 고통"으로 이긴다 (제일 강력)** — 카톡은 3주 뒤 그 사진 못 찾음(묻힘/만료). 앱은 현장별로 남아 바로 찾음. 이 검색 우위가 반복되면 알아서 이주함.
2. **어차피 여는 앱에 얹는다** — 일 배정·오늘 일정 보러 어차피 앱을 열면, 그 김에 올리기가 옆에 있음. 사진이 단독으로 카톡과 싸우지 않고 이미 열린 앱에 업힘.
3. **업로드를 "일의 일부"로 묶는다** — 사진이 있어야 완료 처리/정산/일당 확인. 추가 귀찮음이 아니라 자기가 챙길 일이 됨.

강제가 필요하면 "잔소리"가 아니라 **권한자(대표·사수)의 이유 있는 규칙** 딱 한 지점에만: "정산/증빙 사진은 앱에 올린 것만 인정" (동기=돈·증빙에 연결). **배타적 X** — 앱에서 올리면 카톡으로도 나가게(공유 시트) → 처음엔 병행 → 무게중심 이동.

> 목표는 100% 카톡 대체가 **아님.** "원본 보관·현장별 정리·나중에 찾기"만 앱이 가져와도 승리.

### 올리기 진입점 — 딱 2곳 ✅
> 프로젝트 탭(목록)엔 넣지 않음. 목록에서 올리면 "어느 현장?"을 물어야 해서 카톡과 같은 방-고르기 마찰 발생 → 자동 매칭 이점이 사라짐. **올리기는 "현장이 이미 정해진 곳"에만 둔다.**

1. **프로젝트(현장) 상세 — 그 현장 사진첩** : 확실한 홈, 100% 자동 태깅.
2. **홈의 큰 `📷 사진 올리기` 버튼** : 접근은 여기서 빠르게.

> ⚠️ **"앱이 어느 현장인지 안다"는 가정 금지.** 작업자가 하루에 여러 현장 도는 게 흔함 → 자동 가정하면 **엉뚱한 현장에 사진 꽂힘.** 선택은 없앨 수 없고, 대신 **한 탭으로 실수 없게** 만든다.

**현장 선택 UX (선택을 없애는 게 아니라 가볍게):**
- 사진 버튼 → **오늘/최근 다닌 현장만 큰 칩으로**(회사 전체 X, 보통 2~3개) → 한 탭
- **직전에 올린 현장을 기본값** (한 현장 여러 장 찍을 때 매번 안 고르게)
- **한 번 고르면 세션 동안 고정** = 배치 모드 (10장 찍어도 선택 1번)
- **오태깅해도 복구** — 사진을 다른 현장으로 이동 (카톡은 오발송 복구 불가)

**속도로 카톡을 압도하진 못함(정직).** 선택 한 탭이 껴서 대충 비슷:
| 카톡 | 앱 |
|---|---|
| 열기 → **방 선택** → 카메라 → 촬영 → 전송 | 열기 → 사진 버튼 → **현장 칩 한 탭** → 촬영 → 끝 |

→ **진짜 이기는 축은 속도가 아니라:** 관련 현장만 짧게(오발송↓) · 틀려도 이동 복구 · **현장별로 남아 나중에 찾기(핵심)** · 현장 하나=관련자 자동 공유. 서사는 **"카톡보다 빠르다"가 아니라 "안 틀리고, 나중에 찾을 수 있다".**

역할 분리: **프로젝트 탭 = 현장 둘러보고 관리**, **올리기 = 홈 버튼 + 현장 상세.**

### 기술 메모
- **카메라는 폰 기본 카메라 연동** — `<input type="file" accept="image/*,video/*" capture="environment">`. 버튼 누르면 OS 카메라 앱이 열리고 찍으면 앱으로 돌아옴. (HTTPS 필요 — REVAMP Caddy로 충족.)
  - 앱 안 카메라 직접 구현(`getUserMedia`)은 **안 씀** — 복잡+iOS PWA에서 잘 깨짐. 아저씨·안정성 우선엔 과설계.
  - **동영상도 OS 카메라가 처리** → 앱에 사진/동영상 모드 토글 안 만들어도 됨.
  - 트레이드오프: 기본 카메라는 보통 **한 장씩** 돌아옴 → 여러 장은 "찍고 돌아와 또 찍기" 또는 **앨범 다중선택(`multiple`)**. 현장 몰아찍기엔 앨범 선택이 편할 수 있음.
  - "촬영 중 현장 이름 오버레이"는 못 함(OS 화면이라) → **안전장치는 앞(현장 선택)+뒤(확인 화면)로 대체.**
- S3 저장 + 썸네일 생성, 용량 관리(원본/썸네일 분리), 영상 용량·수명주기 정책 검토(**비용 정책은 나중에**).

| 브라우저 | 자유 | **반드시 사파리** (크롬·카톡 인앱브라우저는 설치/푸시 제약) |

실무 대응:
- **아이폰 설치가 관문** — 첫 설치를 대표/사수가 도와주거나, 앱에 "아이폰 설치법" 그림 안내.
- 아이폰은 **사파리로 열기** 안내(카톡 인앱브라우저·크롬 X).
- 푸시 쓸 거면 아이폰은 최신 iOS+설치 필수. 구형 아이폰은 **카톡 병행이 알림 백업.**
- REVAMP-PLAN Phase 1에 iOS 메타태그+manifest 이미 포함 → iOS 대응 계획엔 반영됨.

## 6. 화면별 기능 핏 — 모바일에서 손질할 것 🤔

> 데스크톱엔 있는데 모바일로 옮기면 삐걱대는 것들. 주말 기능명세에서 화면별로 확정.

| 기능 | 걸리는 점 | 손질 방향(예시) |
|---|---|---|
| **파일/사진(S3)** | 폴더/표 기반, PC 업로드 전제 | → **섹션 5**로 격상. 현장별 사진첩·큰 촬영버튼·공유시트 (핵심 기능) |
| **일정/프로젝트 생성 폼** | 필드 많은 다단 폼 | 한 화면 한 묶음, 스크롤/단계형 시트, 필수만 먼저 |
| **캘린더 월간 편집** | 좁아서 편집 어려움 | 조회는 점(dot), 생성/수정은 시트로 분리 |
| **거래처 상세** | 정보량 많음 | **전화/문자 바로가기 버튼**, 접이식 섹션 |
| **목록 + 페이지네이션** | 페이지 버튼 손가락에 안 맞음 | 무한 스크롤 or "더 보기" 버튼 |
| **권한별 차이** | 일반 사원은 조회 위주 | (권한 완화 후) 역할별 홈·버튼 노출 조정 |

## 7. 주말 기능명세 진행 방법 ✅

목업을 화면 목록 삼아 **화면 하나 = 기능명세 한 꼭지**로 훑는다. 각 화면마다 4칸 채우기:
1. 이 화면에서 뭘 하나 (기능 목록)
2. 데이터 — 뭘 보여주고 뭘 입력받나 (필드)
3. 모바일에서 바뀔 것
4. 권한 — 직원 / 대표 차이

화면 체크리스트:
- [ ] 로그인 (목업에 아직 없음)
- [ ] 홈/대시보드 (역할별 분기?)
- [ ] 일정 목록 (리스트/캘린더)
- [ ] 일정 상세
- [ ] 일정 생성/수정 (바텀시트)
- [ ] 프로젝트 목록
- [ ] 프로젝트 상세
- [ ] 거래처 목록·상세
- [ ] **현장 사진첩** (사진/영상 — 핵심, 섹션 5)
- [ ] 파일함 (문서류 — S3)
- [ ] 사원 관리 (대표)
- [ ] 더보기/설정

## 8. 열린 질문 ❓

- **용어**: 현장에서 실제로 "프로젝트"라 부르나, **"현장"** 이라 부르나? (아저씨 입에 붙은 단어로 라벨링 — `프로젝트→현장`, `담당자→현장 대장` 등)
- **사수가 맡는 현장 수**: 보통 한 번에 1개인가, 여러 현장 동시인가? (홈을 "내 현장" 하나로 볼지, 현장 목록으로 볼지 갈림)
- **역할별 홈을 실제로 나눌지**, 아니면 하나로 두고 요소만 노출/숨김 할지
- 알림(푸시) 범위 — 뭐까지 밀어줄지 (오늘 일정, 배정 알림, 변경 알림, **새 현장 사진**…)
- **사진을 어디에 붙이나** — 현장(프로젝트)에만? 일정에도? 특정 대상 없는 자유 사진첩도 둘지?
- **사진을 하단 탭으로 승격할지** — 카톡만큼 자주 쓰면 '더보기' 안이 아니라 탭에 노출하는 게 나을 수 있음
- 사진/영상 **용량·보관 정책** — 원본 유지 기간, 영상 크기 제한 (S3 비용 연관)
